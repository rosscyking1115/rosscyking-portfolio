## What I built

A release-readiness system for AI-agent changes. Before a changed agent, prompt, model, or tool policy ships, it **replays known incidents, applies policy-as-code gates, and produces ship / warn / block evidence** — so a regression becomes a failed gate, not a production incident.

The project grew out of a public synthetic evaluation lab, which is now the benchmark and evidence layer underneath the gates. All data is fully synthetic (runbooks, tickets, teams, procedures, metrics) so the whole thing can be inspected and extended safely.

## Why it matters

Agent changes are hard to sign off by eye. This turns "is it safe to ship?" into something measurable: does the change stay grounded, refuse unsafe requests, gate risky tool calls, leave an audit trail, and pass replay of past incidents? The output is a reproducible evidence artifact a reviewer can act on.

## Incident replay (the first gate)

The first module turns redacted synthetic incidents into regression fixtures. **8 seeded incidents are replayed on every change with a 100% closure rate and 0 replay must-not violations**, each producing a release gate decision and an incident memo. Policy-as-code gates then turn the evaluation evidence into a single ship / warn / block call.

## Evaluation evidence (the layer underneath)

The controlled benchmark covers 358 synthetic golden cases and 60 red-team cases across 24 runbook sections and 180 synthetic tickets. Two compact public RAG tracks — 160 TechQA and 80 WixQA cases — are evaluated separately to test the retrieval harness beyond self-contained synthetic data.

### Retrieval

Five retrievers evaluated at hit rate@3 and citation coverage. The hybrid and local vector retrievers reach 100% hit rate@3 on the current benchmark. The honest caveat: the local embedding store uses deterministic feature hashing, not a provider-backed model — comparing against a real embedding API is the named next step.

### Safety and red-team

60 red-team cases covering prompt injection (from user prompts and from retrieved documents), context leakage, weak-evidence requests, excessive-agency attempts, access escalation, and tool misuse. The safety classifier reaches 90.91% recall with 0 high-severity false negatives on the current challenge set.

### Tool governance

Read-only tools throughout, plus approval-gated mock side-effecting calls: 100% block rate on unapproved side effects, and 100% of approval events written to the audit log.

### Observability

Trace IDs, audit events, monitoring snapshots, a local span timeline viewer, and an OTLP/HTTP export preview — enough trace and audit evidence for a change to be reviewed, not just trusted.

## Benchmark transparency

The dataset profile is published alongside the results because high scores on synthetic cases only mean something when the benchmark mix is visible: 102 manually authored golden cases (28.5% manual share), 70 expected abstention cases, 46 noise types, 22 task types. The most important gap — templated cases — is documented rather than hidden.

## Delivery

- Public Streamlit dashboard with live benchmark and gate readouts
- GitHub Pages report site with the full evaluation report, PDF, and dataset profile
- FastAPI service with prediction and retrieval endpoints
- Docker / Compose for local full-stack runs
- CI running lint, tests, deterministic report regeneration, and Docker build on every PR

## Honest limits

The benchmark is synthetic and partly templated — scores are engineering checks, not production performance claims. Structured extraction uses deterministic pattern matching, not LLM extraction. Each limit is documented in the repo rather than presented as equivalent to a production deployment.
