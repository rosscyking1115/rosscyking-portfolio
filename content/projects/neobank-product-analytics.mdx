## What I built

A full-stack product analytics portfolio project that simulates the end-to-end workflow of a Product Data Scientist in a modern fintech squad — from synthetic data generation through to PM-ready decision memos and a live Streamlit dashboard.

The project is built around a modern fintech stack: DuckDB as the warehouse, dbt for governed metric transformations, Python for modelling and experimentation, and Marimo notebooks for the analytical narrative.

### Synthetic data generation

A reproducible generator produces realistic neobank event data — users, transactions, sessions, feature events (Savings Pots, Salary Sorter, referrals), support contacts, experiment assignments, and regional daily signups — at scales from 5,000 (CI) to 50,000 users (portfolio). All raw data is gitignored and re-creatable from configs.

### dbt metrics layer

A trusted metrics layer built in dbt covers:

- **D7 activation** — first card transaction within 7 days of signup
- **W4 retention** — activated users transacting in week 4
- **Feature adoption** — Savings Pots, Salary Sorter, referrals, and related product use
- **WAU and transaction frequency**
- **CLV proxy** — simulated 12-month net revenue per user
- **Experiment user metrics** — CUPED-ready pre-experiment covariates
- **Guardrails** — support contact rate, complaint load, fraud flags, crash rate, vulnerable-customer impact, and fair-value signals

### A/B experimentation (personalised onboarding test)

Full experiment analysis with: sample-ratio mismatch (SRM) check, CUPED variance reduction, guardrail metrics, power analysis, heterogeneous treatment effects, and a structured launch recommendation memo. CUPED is implemented from first principles rather than via a library.

### Activation decisioning model

A calibrated binary classifier for D7 activation, with: proper train/test split, calibration curve, SHAP explainability, segment-level fairness checks, and a model card documenting customer-outcome guardrails. Designed to the standard expected of a model that could influence product targeting.

### Regional referral incrementality (geo-lift)

Estimates whether a referral incentive programme drove incremental signups in treated regions — not just correlated ones. Methods: difference-in-differences with parallel trends tests, synthetic control, spillover checks, placebo regressions, and embedded ground-truth recovery to validate the estimator against known lift.

### Delivery layer

A Streamlit dashboard reads dbt marts from DuckDB and displays product health, onboarding A/B results, and referral geo incrementality. If no database exists on first load, it auto-generates a 5,000-user demo dataset and builds the dbt marts — zero-setup for reviewers. One-page decision memos for both the A/B test and the geo experiment are written for a product manager audience.

## Why this project exists

This was built to demonstrate the full scope of a product data scientist role — not just modelling, but the surrounding rigour: governed metrics, experiment design, causal thinking, guardrails, and communication. Each chapter has an honest limits section documenting what the synthetic setup can and cannot show.

## Stack choices

DuckDB was chosen over a hosted warehouse because it keeps the project fully local and reproducible without cloud credentials. dbt enforces a clean interface between raw data and analytics — the same discipline a real team would expect. Marimo notebooks are used for the analytical narrative because they re-run top-to-bottom without hidden state.
